# Research Library — Methods Shell

An automated pipeline that discovers, summarizes, and files AI/ML papers into an indexed, interconnected Obsidian knowledge base. The goal is consistent formatting and dense linking so Obsidian's graph view surfaces meaningful connections across topics.

**This repo is the machinery without the collection.** It carries the full toolchain — the Claude Code skills, the Python scripts, the tests, and the design docs — but ships with an empty `Library/`. It is meant to be read as a working example of the pipeline, or cloned and pointed at a library of your own. My actual filed papers live in a separate private vault; nothing in this repo depends on them.

`Library/`, `discovery/`, and `staging/` exist here but are empty. They are the pipeline's read and write targets, and the scripts expect them to be present — so they're checked in with a `.gitkeep` rather than left to be created by hand.

## Layout

| Folder | What's in it |
|--------|--------------|
| `Library/` | **Empty here.** In a populated vault this is the canonical knowledge base, organized by topic (not year). Each topic folder holds a semantic `MOC.md`, a machine-readable `index.json` of filed papers, and one note per paper. Create `Library/<Topic>/` to start one. |
| `discovery/` | **Empty here.** Fills with daily auto-generated digests of recent papers, filtered by the Library topics. |
| `staging/` | **Empty here.** Fills with unfiled summaries, pipeline worklists, and discovery picks awaiting filing. |
| `scripts/` | Automation: daily discovery job, index rebuilder, discovery review UI. |
| `.claude/skills/` | The skills that drive the pipeline (discovery → summary → filing). |
| `docs/` | Code-development standards and execution plans. |
| `tests/` | Self-checking tests for the index rebuilder and review server. No pytest needed — run each file directly. |

## The pipeline

Papers flow through four stages, each backed by a Claude Code skill:

1. **Discover** (`paper-discovery`) — A daily `launchd` job fetches recent HuggingFace papers and filters them by topic (BM25 keywords + semantic similarity against each topic's `MOC.md`), writing a digest to `discovery/`.
2. **Triage** (`scripts/launch_review.sh`) — A local-only web UI for visually picking papers from the digests; selections land in `staging/`.
3. **Summarize** (`arxiv-summary`) — Generates a Heilmeier-style summary with YAML frontmatter from an arXiv link or PDF.
4. **File** (`library-filing`) — Moves a summary into the right `Library/<Topic>/`, stamps its `topic`, and registers it in that topic's `index.json`.

For bulk work, `pipeline-orchestrator` runs all four stages across many papers from a single prompt (parallel summaries, serial filing).

## Common tasks

```bash
# Run the tests — these pass against an empty Library, no pytest required
python tests/test_rebuild_index.py
python tests/test_review_server.py

# Scan for new papers (inside a Claude Code session)
/paper-discovery                 # or: /paper-discovery days=3

# Launch the discovery review UI on localhost
bash scripts/launch_review.sh    # optional [port], default 8000

# Rebuild every Library/<Topic>/index.json from note frontmatter
source researchEnv/bin/activate
python scripts/rebuild_index.py            # add --check to report drift without writing
```

With an empty `Library/`, `rebuild_index.py` is a no-op that renders an empty overview, and the discovery skill will find no topics to filter against — both are expected until you add a `Library/<Topic>/` with an `MOC.md`.

## Conventions (quick reference)

- **New notes** go in `Library/<Topic>/`, named `YYYY-MM-DD Paper Name.md`, opening with a YAML frontmatter block. Never create notes in the year-based folders.
- **Linking:** `[[Paper Name]]` for notes, `[[concept]]` for concept notes. A note is registered in `index.json`, not by editing the MOC.
- **MOCs are semantic only** — topic scope + curated concept links; they do not enumerate papers.
- **Images** live in `Images_YYYY/` (not present in this repo — create as needed) and embed with `![[filename.png]]`; they resolve vault-wide by filename, so moving (not renaming) them is safe.

## Environment

Python tooling expects a `researchEnv/` virtualenv at the repo root. It is gitignored, so this repo does not ship one — create it before running the scripts:

```bash
python3 -m venv researchEnv
source researchEnv/bin/activate
pip install pyyaml python-docx sentence-transformers
```

`pyyaml` is what the index rebuilder and review server need; `python-docx` is only for `scripts/md_to_docx.py`, and `sentence-transformers` only for the semantic-similarity step of paper discovery. The two test files need nothing beyond `pyyaml`.

`.claude/settings.local.json` is likewise gitignored — Claude Code will create it on first run.
